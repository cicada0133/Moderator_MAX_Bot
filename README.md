# MAX chat moderation bot

Бот удаляет новые сообщения в MAX-чате, если в тексте найден мат. Логика работает с событиями `message_created` и удаляет сообщение через `DELETE /messages`.

В личных сообщениях бот отвечает только администраторам. Для остальных пользователей доступна только команда `/id`; любые другие сообщения и команды в ЛС бот молча игнорирует.

## Описание для карточки бота

```text
Бот-модератор чата. Удаляет сообщения с ненормативной лексикой.

В личных сообщениях бот отвечает только администраторам. Если вам нужен доступ к управлению ботом, отправьте команду /id и обратитесь к администратору чата.
```

## Что нужно в MAX

1. В настройках бота разрешить добавление в групповой чат.
2. Добавить бота в нужный чат.
3. Сделать бота администратором чата.
4. Выдать права, достаточные для чтения всех сообщений и удаления сообщений.

Без прав администратора бот не будет получать события из группового чата, а без права удаления API вернёт ошибку при `DELETE /messages`.

## Первый запуск через Long Polling

Long Polling подходит для разработки и проверки. Для production в документации MAX рекомендуется Webhook.

```powershell
Copy-Item .env.example .env
notepad .env
npm install
npm run check
npm start
```

Для безопасной первой проверки поставьте в `.env`:

```dotenv
DRY_RUN=true
MODERATION_NOTIFY=true
MODERATION_WARNING=Тест: нашёл запрещённое слово "{token}", правило "{reason}". Пока DRY_RUN=true, сообщение не удаляю.
```

Так бот будет писать в консоль, какие сообщения он удалил бы, но не будет вызывать удаление.

Для боевого режима:

```dotenv
DRY_RUN=false
MODERATION_NOTIFY=true
MODERATION_WARNING={user}, ваше сообщение удалено: в чате запрещена ненормативная лексика. Давайте жить дружно 🙂
```

В уведомлении доступны шаблоны `{user}`, `{username}`, `{userId}`, `{token}`, `{reason}` и `{action}`. Для обычной работы лучше не показывать `{token}` в чате, чтобы не повторять удалённую лексику.

## Production через Webhook

Webhook должен быть публично доступен по HTTPS на порту 443. Локально приложение слушает `WEBHOOK_PORT`; обычно перед ним ставят nginx, Caddy, Traefik или другой reverse proxy с HTTPS.

1. Заполните в `.env`:

```dotenv
WEBHOOK_PUBLIC_URL=https://your-domain.ru/max/webhook
WEBHOOK_SECRET=change_me_12345
WEBHOOK_PATH=/max/webhook
```

2. Запустите сервер:

```powershell
npm run start:webhook
```

3. В отдельном терминале подпишите бота на webhook:

```powershell
npm run webhook:subscribe
```

После активной webhook-подписки Long Polling не работает. Для локальной разработки используйте что-то одно.

## Настройка фильтра

Встроенный фильтр ловит основные русские корни мата и простую маскировку внутри слова: точки, цифры и похожие латинские буквы.

Посмотреть встроенный словарь можно локально:

```powershell
npm run dictionary:show
```

Дополнительные слова можно добавить без изменения кода:

```dotenv
CUSTOM_BAD_WORDS=оскорбление1,оскорбление2
```

Исключения добавляются так:

```dotenv
ALLOW_WORDS=слово1,слово2
```

## Команды бота

Чтобы менять словарь через MAX, сначала напишите боту:

```text
/id
```

Бот вернёт ваш `user_id`. Добавьте его в `.env`:

```dotenv
BOT_ADMIN_IDS=123456789
CUSTOM_DICTIONARY_PATH=data/custom-dictionary.json
CUSTOM_ADMINS_PATH=data/admins.json
CUSTOM_SANCTIONS_PATH=data/sanctions.json
```

После изменения `.env` перезапустите бота. Команда `/id` доступна всем, остальные команды доступны только администраторам бота.

Команды администратора:

```text
/badwords
/banword слово
/unbanword слово
/allowword слово
/unallowword слово
/admins
/addadmin user_id
/removeadmin user_id
/ban user_id 30m|1d|7d|forever
/unban user_id
/bans
```

Команда `/badwords` показывает размер встроенного словаря и пользовательские добавления. Полный встроенный словарь в чат не выводится, чтобы не засорять переписку.

Команда `/admins` показывает администраторов человекочитаемо, если бот уже видел профиль пользователя:

```text
Администраторы бота:
- Мария (@maria, 123456789)
- Павел Лебединский (987654321)
```

В MAX имена в этом списке отправляются как кликабельные упоминания пользователя. Бот запоминает имя и `@username` из сообщений администраторов и из отправленных контактов. Если профиль ещё неизвестен, в списке останется только `user_id`.

Команда `/addadmin` добавляет администратора в локальный файл `data/admins.json`, поэтому перезапуск после неё не нужен. Администраторы из `.env` считаются базовыми: командой `/removeadmin` их удалить нельзя, чтобы случайно не потерять доступ к управлению ботом.

Администратор также может отправить боту контакт пользователя. Если MAX передаст в карточке `user_id`, бот покажет этот id и даст кнопки:

```text
Добавить админом
Убрать админа
Показать админов
```

Если в контакте окажется только телефон или vCard без MAX `user_id`, бот напишет, что id не найден. Тогда попросите пользователя отправить боту `/id` и добавьте его через `/addadmin user_id`.

## Soft-ban

Soft-ban не удаляет пользователя из чата и не запрещает ему писать средствами MAX. Бот просто хранит временную санкцию и автоматически удаляет новые сообщения пользователя, пока ban активен.

Soft-ban всегда привязан к конкретному чату: один и тот же пользователь может быть ограничен в одной группе и свободно писать в другой.

Администраторов бота нельзя отправить в soft-ban, чтобы случайно не потерять управление. При этом обычная модерация продолжает действовать: если администратор пишет мат, бот удалит сообщение и отправит предупреждение как обычно.

Команды администратора в нужном чате:

```text
/ban user_id 30m
/ban user_id 1d
/ban user_id 7d
/ban user_id forever
/unban user_id
/bans
```

Если администратор отправит контакт пользователя в группе, бот покажет кнопки:

```text
Бан 30 минут
Бан 1 день
Бан 7 дней
Бан навсегда
Снять бан
Показать баны
```

В ЛС эти кнопки не показываются, потому что soft-ban должен знать, в каком именно чате действует ограничение.

Если фильтр слишком строгий или мягкий, правьте правила в `src/profanity.js` и запускайте:

```powershell
npm test
```

## План версии 2

Текущая версия проверяет сообщения по словарю, нормализации символов и regex-правилам. Этого достаточно для базовой антимат-модерации, но такой подход не понимает контекст и не ловит сложные нарушения без прямых ключевых слов.

Версия 2 должна добавить ИИ-слой модерации поверх текущего фильтра:

1. Быстрый локальный фильтр остаётся первым уровнем.
   Он сразу удаляет очевидный мат и простые обходы без обращения к ИИ.

2. ИИ-проверка включается для неоднозначных сообщений.
   Модель должна оценивать контекст, намерение, риск и категорию нарушения, а не только отдельные слова.

3. Категории v2:
   - скам, фишинг, обманные ссылки, просьбы перевести деньги или выдать коды;
   - спам и массовые рекламные сообщения;
   - угрозы, травля, агрессивные персональные оскорбления;
   - запрещённый сексуальный контент;
   - материалы сексуальной эксплуатации несовершеннолетних, CSAM/ЦП;
   - инструкции по обходу правил чата или модерации.

4. Решения по уровню уверенности:
   - высокая уверенность: удалить сообщение и уведомить автора;
   - средняя уверенность: скрыть/пометить для ручной проверки, если MAX даст подходящий сценарий;
   - низкая уверенность: не удалять, только логировать категорию и причину для анализа.

5. Для особо опасных категорий, включая CSAM/ЦП:
   - не пересылать и не цитировать содержимое в уведомлениях;
   - хранить минимум данных в логах;
   - удалять сообщение при высокой уверенности;
   - предусмотреть отдельный сценарий эскалации владельцу чата или ответственному администратору.

6. Админские настройки:
   - команды для включения/выключения ИИ-модерации;
   - пороги уверенности по категориям;
   - белый список доменов и доверенных участников;
   - журнал последних срабатываний без публикации запрещённого текста в чат.

7. Технический план:
   - вынести модерацию в общий пайплайн `rules -> ai -> action`;
   - добавить отдельный модуль `ai-moderation`;
   - сделать dry-run режим для ИИ-решений;
   - покрыть тестами категории, пороги и fallback при ошибке API;
   - не блокировать чат, если ИИ-сервис временно недоступен.
